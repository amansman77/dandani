# ADR-PROD-001 Action Flow MVP

## 상태
제안

---

## 맥락

Dandani V2는 다음과 같이 정의된다:

> 개인의 감정 상태와 행동 반응을 기반으로  
> 지금 실행 가능한 작은 행동을 제안하는 개인화 에이전트

하지만 정책 수준의 정의만으로는 실제 서비스가 어떻게 동작하는지 명확하지 않다.

따라서 MVP에서는 복잡한 기능을 배제하고,  
사용자가 실제로 경험하게 될 하나의 완결된 행동 흐름을 정의한다.

> 핵심:  
> 사용자는 무엇을 입력하고, 무엇을 받고, 무엇을 하게 되는가?

---

## 결정

Dandani V2 MVP는 다음 단일 흐름을 구현한다:

> 현재 상태 입력 → 원하는 상태 입력 → 행동 제안 → 행동 수행 → 결과 선택 → 느낌 기록 → 회고 → 데이터 저장

이 흐름은 항상 동일하며,  
각 흐름은 하나의 행동과 하나의 결과를 남긴다.

---

## 1. 시작 (Greeting)

단단이가 사용자에게 말을 건다.

> 왔구나. 지금 어떤 상태야?

---

## 2. 현재 상태 입력

사용자는 현재 상태를 한 줄로 입력한다.

예:

> 피곤하고 아무것도 하기 싫어.

---

## 3. 원하는 상태 입력

단단이가 묻는다.

> 그럼 지금은 어떻게 되고 싶어?

사용자는 원하는 상태를 입력한다.

예:

> 조금이라도 해냈다는 느낌이 있었으면 좋겠어.

---

## 4. 행동 제안

단단이는 현재 상태와 원하는 상태를 기반으로  
하나의 작은 행동을 제안한다.

> 좋아. 지금 상태에서 크게 바꾸려 하지 말고, 딱 하나만 해보자.  
> 미뤄둔 일 하나를 끝내려 하지 말고, 딱 3분 동안 열어만 보기.

행동 조건:

- 하나의 행동만 제안
- 10분 이내
- 즉시 시작 가능
- 완료 기준이 명확

---

## 5. 행동 수행

사용자는 실제로 행동을 수행한다.  
(이 구간은 시스템 개입 없음)

---

## 6. 결과 선택

행동 후 단단이는 묻는다.

> 어땠어?

사용자는 다음 중 하나를 선택한다.

- 못했어
- 조금 했어
- 해냈어

---

## 7. 내부 해석 (시스템 처리)

사용자의 선택을 내부적으로 다음 데이터로 변환한다.

| 사용자 선택 | started | completed |
|------------|--------|----------|
| 못했어 | false | false |
| 조금 했어 | true | false |
| 해냈어 | true | true |

---

## 8. 느낌 기록

단단이가 묻는다.

> 하고 나니까 어땠어? 한 줄로 남겨줄래?

사용자 입력 예:

> 시작은 했는데 금방 멈췄어. 그래도 조금 덜 찝찝해.

---

## 9. 단단이 회고

단단이는 결과를 짧게 정리한다.

> 오늘 너는 피곤한 상태에서도 완벽하게 하려 하지 않고 시작하는 쪽을 선택했어.  
> 오늘의 단단함은 끝낸 데가 아니라 시작한 데 있어.

---

## 10. 다음 연결

단단이는 흐름을 마무리하며 다음을 암시한다.

> 다음에도 크게 바꾸려 하지 말고, 지금 상태에서 할 수 있는 가장 작은 한 걸음을 같이 찾아보자.

---

## 11. 저장 데이터

하나의 흐름이 끝나면 다음 데이터를 저장한다.

```json
{
  "current_state": "피곤하고 아무것도 하기 싫어",
  "desired_state": "조금이라도 해냈다는 느낌",
  "suggested_action": {
    "type": "Start Action",
    "description": "딱 3분 동안 미뤄둔 일을 열어보기",
    "estimated_minutes": 3,
    "completion_condition": "3분 동안 작업을 열고 손을 대면 완료"
  },
  "result": "조금 했어",
  "started": true,
  "completed": false,
  "after_feeling": "시작은 했는데 금방 멈췄어. 그래도 조금 덜 찝찝해.",
  "pattern_note": "낮은 진입 장벽에서는 시작 가능하지만 유지 어려움",
  "next_hint": "더 짧고 종료 조건이 명확한 행동이 적합"
}
```

---

## 근거

1. 사용자 경험을 단순화하여 실행 가능성을 높이기 위해
2. 행동 결과를 하나의 선택으로 입력받아 UX 부담을 줄이기 위해
3. 내부적으로는 시작/완료 데이터를 분리하여 학습 가능하게 하기 위해
4. 작은 행동 반복을 통해 장기 변화를 유도하기 위해
