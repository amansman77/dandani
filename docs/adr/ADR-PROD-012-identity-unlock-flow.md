# ADR-PROD-012 Identity Dandani Unlock Flow

## 상태
제안

---

## 맥락

ADR-PROD-011 Identity Collection & Slot System에서  
단단이는 "수집 가능한 정체성"으로 정의되었다.

하지만 현재 구현은 다음과 같은 문제를 가진다:

- "나만의 단단이 만들기" 버튼이 DONE 화면에만 노출됨
- 조건(3회 수행)을 사전에 알 수 없음
- 사용자는 목표를 인지하지 못한 채 지나칠 수 있음

---

👉 문제 정의:

> ❗ 단단이 생성이 "숨겨진 기능"으로 작동한다  
> ❗ 사용자에게 기대감이나 동기 유도가 없다

---

👉 핵심 질문:

> "사용자가 단단이를 '만들고 싶다'고 느끼게 만들려면 어떻게 해야 하는가?"

---

## 결정

Dandani는  
"나만의 단단이 만들기"를 단순 기능이 아니라  
진행도 기반 Unlock Experience로 제공한다.

---

## 핵심 정의

### Unlock Experience

> 사용자가 행동을 반복하며 특정 조건에 도달했을 때  
> 새로운 기능(단단이 생성)이 자연스럽게 열리는 경험

---

## 1. Unlock 조건

```
started = true인 action_flow 누적 3회 이상
```

### 결과 기준

| 선택 | 카운트 |
|------|--------|
| 해냈어 | 포함 |
| 조금 했어 | 포함 |
| 못했어 | 제외 |

👉 이유:

> 단단이는 성취가 아니라 "한 걸음"을 기록하는 시스템이기 때문

---

## 2. Unlock 단계 설계

### Stage 0: 초기 (0~1회)

```
(노출 없음)
```

---

### Stage 1: 진행 중 (1~2회)

```
지금까지 한 걸음이 쌓이고 있어.
3개가 모이면 너만의 단단이를 만들 수 있어.
```

👉 위치: DONE 화면, 기록 탭

---

### Stage 2: 직전 (2회)

```
한 걸음만 더 하면 너만의 단단이를 만들 수 있어.
```

👉 기대감 형성

---

### Stage 3: Unlock (3회 이상)

```
호성, 이제 너만의 단단이를 만들 수 있어.
지금까지의 한 걸음들이 쌓였어.
```

👉 CTA:

```
[ 나만의 단단이 만들기 ]
```

---

## 3. DONE 화면 역할 변경

기존:
- 결과 표시
- 종료

변경:

> DONE 화면 = 다음 행동으로 이어지는 허브

구성:

```
결과 피드백 → 진행도 안내 → Unlock 상태 안내 → CTA (있을 경우)
```

---

## 4. CTA 정책

### 조건 미충족
- 버튼 없음
- 진행도만 표시

### 조건 충족
- CTA를 메인 버튼으로 배치
- "다시 시작하기"보다 우선순위 높음

---

## 5. UX 원칙

### 1) 숨기지 않는다
- Unlock 조건은 사전에 인지 가능해야 한다

### 2) 부담을 주지 않는다
- 강요 ❌
- 자연스러운 흐름 ⭕

### 3) 의미를 전달한다
```
단단이 생성 = 행동의 결과
```

---

## 6. 금지 사항

- 갑작스러운 기능 등장
- 조건 미설명
- 단순 보상형 UX (게임처럼 보이게 하는 것)

---

## 7. 기대 효과

- 행동 반복 유도
- 리텐션 증가
- 단단이 생성 기능 인지도 상승
- 자연스러운 전환(Conversion) 흐름

---

## 8. 향후 확장

- Unlock 애니메이션
- "단단이 생성까지 남은 거리" 시각화
- 단계별 메시지 개인화

---

## 한 줄 정의

> 단단이는 주어지는 것이 아니라  
> 👉 한 걸음이 쌓였을 때 자연스럽게 열리는 것이다

---

## 한 줄 정리

> 나만의 단단이 만들기는 기능이 아니라  
> 👉 사용자가 만들어낸 결과를 열어보는 순간이다
