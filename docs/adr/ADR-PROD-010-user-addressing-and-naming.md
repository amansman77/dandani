# ADR-PROD-010 User Addressing & Naming

## 상태
제안

---

## 맥락

Dandani V2는 개인화 행동 에이전트를 목표로 한다.

현재까지:
- 감정 기반 행동 제안 ⭕
- 패턴 기반 개인화 ⭕
- 개인화 Greeting ⭕

하지만 사용자와의 관계에서 중요한 요소가 빠져 있다:

> ❗ 사용자를 어떻게 부를 것인가

현재는 "너" 또는 생략. 관계 형성이 약하고, 개인화 체감 부족.

---

## 결정

Dandani는 초기 진입 시 자신을 소개하고, 사용자 호칭을 직접 묻는다.
그리고 이후 모든 대화에서 해당 호칭을 사용한다.

---

## 1. 초기 플로우

첫 방문 시:
```
나는 단단이야.
너를 어떻게 부르면 좋을까?
```

사용자 입력: 이름 / 별명 / 아무 입력

---

## 2. 호칭 저장

```json
{
  "user_id": "123",
  "name": "호성",
  "address_style": "neutral"
}
```

---

## 3. 호칭 사용 규칙

**기본:**
```
호성, 지금 어떤 상태야?
```

**자연스러운 생략:** 모든 문장에 넣지 않는다.
```
호성, 오늘 어때?
→ 이후 문장에서는 생략 가능
```

---

## 4. Tone 적용

- 부드럽게 사용
- 과도한 반복 금지
- 감정 강조 시 사용

예:
```
호성, 좀 지친 것 같네.

오늘은 아주 작게 가보자.
```

---

## 5. 선택적 기능

- 호칭 변경 가능
- 호칭 사용 여부 설정 가능

---

## 6. 금지 사항

- 강제 친근함 금지 (예: "야", "친구야")
- 존칭 강제 금지
- 모든 문장에 호칭 삽입 금지

---

## 7. 기대 효과

- 관계 형성 강화
- 개인화 체감 상승
- 감정 몰입 증가

---

## 한 줄 정의

> 단단이는 사용자를 분석하는 존재가 아니라  
> 이름을 부르는 존재다

---

## 이 기능의 진짜 의미

이건 단순 UX 기능이 아니라:
> "단단이가 나를 아는 존재"라는 증거

한 단계 더 가면:
- "호성, 오늘은 조금 피곤해 보이네"
- "호성, 지난번보다 조금 나아졌어"

---

## 한 줄 정리

> 개인화의 시작은 데이터가 아니라  
> 호칭이다
